# 开发模式演进路线

## 这份文档记录什么
这份文档记录的是：这个新项目在开发方式上应该怎么推进、为什么当前仍以对话框模式为主、后面怎样逐步过渡到更成熟的多 Agent 协作模式，以及最终理想状态是什么。

它关注的是“怎么工作”，不是“引擎内部怎么设计”。

---

## 当前结论

当前最稳的开发方式仍然是：

- 对话框模式主力开发

但这里的“对话框模式”不是原地不动，而是要开始补下面这些能力：

- 文件化上下文
- 任务拆解
- 角色分工
- 辅助会话审查

原因是当前我们同时在推进：

- `deep_research_agent` 的主链收敛
- `adaptive-agent-orchestrator` 的引擎起步
- evaluate 能力的未来抽离

如果现在直接整套切到“多模型、多角色、全并行、几乎不自己介入”的模式，风险会偏高：

- 主线容易丢
- 任务边界容易混
- 文件交接机制还没完全稳定
- 自己的调度习惯还没练熟

所以当前最优解不是“维持原样”，也不是“整套照搬朋友的模式”，而是：

- 主力继续用对话框模式
- 但把朋友那种模式里的关键习惯一点点引进来

---

## 现在的工作方式

当前更像：

- 一个人类负责人
- 一个主 AI 会话
- 一步一步推进任务

它本质上更接近：

- 结对开发
- 串行协作
- 逐步验收

在这个阶段，人类主要承担：

- 目标描述
- 中间纠偏
- 每一步验收
- 最终拍板

AI 主要承担：

- 代码生成
- 逻辑解释
- 方案草拟
- 局部修改

这不是落后的模式，只是更偏：

- 可控
- 细粒度
- 低并发

---

## 为什么朋友那种模式听起来更高效

朋友那种模式，本质上不是“终端更高级”，而是：

- 一个总控入口
- 多个角色化 Agent
- 围绕同一个项目目录和状态文件协作

也就是：

- Planner Agent（规划智能体）
- Executor Agent（执行智能体）
- Reviewer Agent（审查智能体）
- Supervisor Agent（监督 / 总控智能体）
- Evaluator Agent（评估智能体）

它们依赖的共享介质不是单一聊天线程，而是：

- 代码仓库
- `PROJECT_STATE.md`
- `CURRENT_TASK.md`
- `REVIEW_NOTES.md`
- 测试结果
- diff
- 任务文档

所以它听起来高效，是因为它更像：

- 小型 AI 团队

而不是：

- 单个聊天助手

---

## 人类总控和总控 Agent 的关系

这里最容易误解的一点是：

- 人类总控
- 总控 Agent（Supervisor Agent）

并不是二选一的关系。

更成熟的系统里，通常会变成：

- 人类保留最高权限
- 但很多日常总控动作，可以委托给总控 Agent 先做

例如下面这些动作：

- 拆任务
- 分角色
- 汇总进展
- 做第一轮 review
- 给出下一步建议

这些可以由 Supervisor Agent 代理完成。

而人类主要保留：

- 最终边界定义
- 风险判断
- 是否接受结果
- 是否回退 / 改策略
- 最终拍板

所以“人类总控”真正意味着：

- 人类拥有最高控制权

不意味着：

- 人类必须亲自执行所有总控动作

---

## 为什么当前不整套切过去

当前还缺的不是工具，而是下面这些稳定习惯：

- 文件化上下文
- 任务拆解
- 角色边界定义
- 并行验收
- 结果整合

如果这些东西还没稳住，就直接切到朋友那种模式，很容易出现：

- 多个会话都在做主决策
- 没有统一状态源
- 结果互相冲突
- 人自己更累

所以当前路线应该是：

- 先稳住主力开发方式
- 再把朋友那种模式里的关键能力一点点引进来

---

## 过渡路线

### 阶段 1：对话框主力 + 文件化上下文
这是当前阶段。

工作方式：

- 主力仍然是对话框模式
- 项目上下文开始显式写进文件

当前至少要稳定维护：

- `PROJECT_STATE.md`
- `PROJECT_LOG.md`
- 必要时新增 `CURRENT_TASK.md`

目标：

- 不再把关键上下文只放在聊天里
- 让项目状态能被回看、被续接、被别的会话读取

### 阶段 2：1 主 1 辅
这是最近可以开始练的模式。

工作方式：

- 一个主会话负责真正推进任务
- 一个辅助会话负责 review、第二意见、风险提醒

这一步开始训练的不是“更多工具”，而是：

- 任务拆分
- 角色边界
- 验收标准

### 阶段 3：1 主 2 辅
等状态文件和分工习惯稳定后，再升级到：

- 主执行
- 辅助审查
- 测试 / 文档 / 评估辅助

这时开始更像一个小型协作组。

### 阶段 4：总控式多 Agent 工作流
这是未来目标，不是当前要求。

理想状态：

- 一个总控入口
- 多个角色化 Agent
- 围绕共享项目状态协作
- Supervisor Agent 先代理大量日常总控动作
- 人类主要保留最高权限、做最终判断和整合

这时人类的角色不再是“每一步都参与的人”，而更像：

- tech lead
- orchestrator
- final decision maker

---

## 最终理想状态

最终理想状态不是“完全不动手”，而是：

- 人类不再盯每一步细节
- AI 不再只是一问一答的助手
- 项目状态被外部化到文件和系统里
- 多个角色化 Agent 能围绕同一项目协作
- 日常总控动作越来越多地由 Supervisor Agent 代理
- 人类主要负责：
  - 拆任务的最终边界
  - 设验收标准
  - 看关键结果
  - 做取舍
  - 最终拍板

也就是说，最终目标不是：

- “更会聊天”

而是：

- “更会组织 AI 团队做事，并把大量中层调度代理给总控 Agent”

---

## 和本项目的关系

`adaptive-agent-orchestrator` 不是朋友那种工作方式本身，而是那种工作方式背后的：

- 调度系统
- 状态系统
- 评估系统
- 回退和降级系统

更直白地说：

- 朋友是在“使用多 Agent 团队模式做项目”
- 我们的新项目是在“构建这种团队模式背后的运行时内核”

而更进一步说，未来这个引擎还应该支持：

- Supervisor Agent 代理部分总控动作
- 人类只在高风险和高杠杆节点介入

所以当前路线不是脱离主线的新尝试，而是在为未来更成熟的工作方式打基础。

---

## 一句话总结

当前路线不是“坚持旧模式”，也不是“立刻模仿朋友全套模式”，而是：

先用对话框模式稳住主线，再把文件化上下文、角色分工、辅助会话和调度思维一点点引进来，最终过渡到“人类保留最高权限、Supervisor Agent 代理大量日常总控动作、多 Agent 团队负责协作执行”的工作方式。
