# 003 - 让 Supervisor 开始理解执行过程

## 问题

当前 `SupervisorAgent` 已经进入 workflow，但它主要只读取最终产物：

- `query`
- `plan`
- `raw_documents`
- `summary`

这会带来一个明显问题：

- 它像一个只看最终答案的主管
- 还不像一个真正理解执行过程的总控层

如果它看不到执行轨迹、重试情况和状态信息，就无法回答：

- 这轮任务是不是很勉强才完成
- 中间有没有出现失败或重试
- 当前系统状态是否稳定

## 决策

本次决定让 `SupervisorAgent` 开始读取最小过程信息：

- `execution_trace`
- `retry_counters`
- `global_step`
- `status`
- `failure_reason`

并在 `supervisor_report` 中新增：

- `process_review`

用于汇总：

- 当前看到的执行步数
- 评估事件数量
- 重试次数
- 是否出现未通过评估
- 当前状态

## 取舍

为什么现在就做这一步：

- `Supervisor` 已经进入 workflow，下一步最值钱的增强就是让它从“只看结果”升级为“开始看过程”
- 这一步能明显增强项目的层级感，让 `Human -> Supervisor -> Worker` 更接近真实总控关系
- 这一步实现成本低，但对项目叙事价值很高

为什么没有直接做完整长期记忆或复杂总控策略：

- 当前阶段最重要的是让 `Supervisor` 对 runtime 有最小过程理解
- 如果现在直接做长期记忆、动态 re-plan、多模型路由，会把问题做大
- 先把“能看过程”做实，再往“能基于过程做更强决策”扩，顺序更稳

## 当前边界

当前 `Supervisor` 仍然只是最小过程感知版本：

- 能读取最近一段执行轨迹
- 能看到重试和失败痕迹
- 能输出过程摘要

但它还不能：

- 主动改写 workflow
- 决定真正的 re-plan
- 调用长期记忆层
- 做多模型路由

## 后续演化

下一步可以继续增强：

1. 让 `Supervisor` 读取更结构化的执行审计信息
2. 让 `Supervisor` 结合失败类型提出更清楚的修正建议
3. 再往后接入长期记忆和更复杂的反馈闭环

## 面试 / 博客表达

这一轮改动可以这样讲：

> 我没有把 Supervisor 停留在“最终多一个 review 节点”的层面，而是让它开始读取运行时轨迹、重试次数和状态信息。这样它第一次具备了最小的过程理解能力，更像一个总控层，而不是只看最终答案的角色。
