# RegressionCompare - 回归对比模块

## 1. 问题

当项目持续迭代时，如何判断一次改动是否引入了性能退化？

痛点：
- 手动对比两次运行结果效率低
- 缺乏统一的退化判定标准
- 无法快速定位退化的具体指标

它在整个系统里的位置：
- 属于 `analyze` 命令的子功能
- 依赖 `outputs/reports/` 和 `outputs/states/` 中的运行产物

---

## 2. 决策

实现 `RegressionCompare` 类，提供运行间对比能力。

核心对象：
- `RegressionCompare` - 对比器主类
- `RegressionReport` - 对比结果报告
- `MetricDiff` - 单指标差异
- `RegressionSignal` - 退化信号枚举

对比维度：
- `steps_executed` - 执行步数
- `failed_evaluations` - 评估失败数
- `retry_count` - 重试次数
- `success_rate` - 成功率
- `confidence` - 置信度

退化阈值：
- `MINOR_THRESHOLD = 0.1` - 10% 变化视为轻微退化
- `MAJOR_THRESHOLD = 0.3` - 30% 变化视为严重退化

---

## 3. 取舍

### 备选方案

1. **基于全量 state diff**
   - 优点：信息完整
   - 缺点：噪音大，难以聚焦关键指标
   - 未采用

2. **基于 report 摘要对比（采用）**
   - 优点：聚焦关键指标，结果清晰
   - 缺点：可能遗漏细节
   - 采用原因：符合"快速定位问题"的目标

### 牺牲

- 不做逐字段全量对比
- 不区分"故意失败测试样本"和"真实退化"

### 换来

- 快速的退化检测
- 清晰的退化信号分类
- 可操作的对比报告

---

## 4. 当前边界

**未覆盖：**
- 智能区分退化原因（代码问题 vs 测试样本变化）
- 跨 workflow 对比
- 历史趋势分析

**保持简单：**
- 只对比同 workflow 的相邻运行
- 只关注 5 个核心指标

## 6. 面试 / 博客表达

### 一句话版本

RegressionCompare 提供运行间自动对比，帮助快速发现性能退化。

### 稍展开版本

RegressionCompare 是一个运行对比工具，通过比较执行步数、失败数、重试次数等关键指标，自动判定是否存在性能退化。它支持单次对比、批量对比和退化筛选，帮助开发者在迭代中快速定位问题。
